Introducción

Noviembre 7, 2022 by Jimmy Castro

Se presentará casos de pruebas referentes a los módulos establecidos. Estos módulos representarán el enfoque y la programación de todas las fases del proyecto, los cuales estarán previamente analizados y así ser requeridos para el sistema de información.

Objetivos

Nuestro plan de pruebas tiene como objetivo orientar el esfuerzo de pruebas, identificando y detallando las pruebas más importantes, para que el equipo pueda enfocarse en su ejecución y pueda responder de forma adecuada a los cambios que tiene el proyecto, para así atender los objetivos de calidad en el desarrollo de sistemas, encargándose de definir aspectos como por ejemplo los módulos o funcionalidades, tipos de pruebas, entornos, recursos asignados, entre otros aspectos.

Tareas

  • Diseñar un caso de prueba
  • Hacer informes de problemas y del caso de prueba
  • Tener un resultado esperado

Audiencia prevista

En la audiencia previste de este plan habrá diferentes grupos potenciales. Que entre ellos están:

  • Equipo de pruebas
    • Supervisar aplicaciones y sistemas de software. Llevar a cabo pruebas de estrés, pruebas de rendimiento, pruebas funcionales y pruebas de escalabilidad. Escribir y ejecutar scripts de prueba. Realizar pruebas manuales y automatizadas.
  • Equipo de desarrollo
    • Este equipo se encarga de desarrollar el producto, organizando y gestionando para conseguir entregar un incremento de software al final del ciclo de desarrollo.
  • Jefe de proyecto
    • Gestiona de manera global el proyecto. Planifica, define los recursos, coordina y supervisa el análisis funcional y orgánico, así como también el desarrollo y la implementación del software.
  • Grupos de aseguramiento de la calidad
    • Este grupo tiene las actividades planificadas y sistemáticas necesarias para aportar la confianza que el software satisfará los requisitos dados de calidad. Este aseguramiento se diseña para la aplicación antes de comenzar a desarrollarla en el que debe tener:
  • Un enfoque de gestión de calidad.
  • Métodos y herramientas de Ingeniería del Software.
  • Revisiones técnicas formales en el proceso del software.
  • Una estrategia de prueba multiescalar.
  • El control de la documentación del software y de los cambios realizados.
  • Procedimientos para ajustarse a los estándares de desarrollo del software.
  • Mecanismos de medición y de generación de informes.

Cuestiones de riesgo

El objetivo de esta disciplina es identificar, abordar y mitigar elementos de riesgo antes de que se conviertan en una amenaza para la ejecución exitosa del proyecto y para el logro de los objetivos planteados.

Caracteristicas a probar

Las características por probar desde el punto de vista del Usuario es que cada uno de estos módulos respondan de manera correctas a sus necesidades como, por ejemplo:

  • Modulo de login
    • El usuario debe hacer el login de manera correcta respetando sus condiciones para así obtener el acceso al sistema
  • Registro de usuario
    • Desde el punto de vista del usuario al llenar el registro con toda la información requerida este registro debe guardarse en la base de datos del sistema para después progresar y loguearse
  • Modulo de consulta
    • Lo que espera el usuario que responda de manera correcta es que cada vez que lo necesite este módulo le permitirá la búsqueda de cada uno de los productos.
  • Modulo de carrito
    • Se espera que el usuario seleccione el producto y estos sean almacenados automáticamente en el carrito

    Caracteristicas que no se van probar

    Desde el punto de usuario del cliente las características que no se van a probar no porque no sean importantes por el funcionamiento sino porque son lo suficiente mente estables y se encuentran bien reguladas por los respectivos técnicos.

Estrategia de pruebas

Noviembre 24, 2022 by Jimmy Castro

Estas estrategias de pruebas de software permiten la ejecución de un sistema con el objetivo principal de encontrar errores presentes en el mismo; así conseguimos reducirlos y mejorar la calidad del sistema, haciéndolo más robusto frente a fallos. Entre estos grupos están los siguientes:

  • Pruebas unitarias
    •

    Es el mecanismo de comprobación del funcionamiento de las unidades de menor tamaño de un programa o la aplicación en específico. Esta estrategia de prueba es ágil para el trabajo del desarrollo, donde se busca ofrecer piezas pequeñas de software en funcionamiento en un corto periodo de tiempo, con el objetivo de aumentar la satisfacción del cliente.

  • Pruebas de integracion
    •

    El análisis de integración de software analiza la integración o relación de entidades, las relaciones que existen entre los componentes del sistema, como sistemas operativos, archivos y dispositivos, o las conexiones entre sistemas. Para lograr resultados óptimos y de alta calidad, es importante que el analista de software sea experto en técnicas de análisis colaborativo.

  • Pruebas de sistema
    •

    Este es el proceso de evaluar y verificar que un producto o aplicación de software hace lo que se supone que debe hacer. Los beneficios de las pruebas incluyen la prevención de errores, la reducción de los costos de desarrollo y la mejora del rendimiento

  • Pruebas de carga
    •

    Las pruebas de carga de definición generalmente se refieren a las pruebas como un subconjunto del proceso de pruebas de rendimiento del software, que normalmente también incluye varios otros tipos de pruebas, como pruebas de esfuerzo, pruebas de remojo, pruebas de picos, pruebas de resistencia, pruebas de volumen y pruebas de escalabilidad, entre otros tipos de pruebas.

  • Pruebas de aceptación de usuario
    •

    Esta prueba del software por parte del usuario o cliente se hace para determinar si puede ser aceptado o no. Esta es la prueba final que se realiza una vez que se completan las pruebas funcionales y de sistema El objetivo principal de esta prueba es validar el software frente a los requisitos comerciales. Esta validación la llevan a cabo los usuarios finales que están familiarizados con los requisitos comerciales.

  • Pruebas de regresion
    •

    Probamos el sistema el cual ha sido analizado previamente para asegurar que no se haya introducido algún tipo de defecto como resultado de cambios realizados.
